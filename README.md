# ROS2 SLAM Engineering Notebook

A public engineering notebook documenting my self-directed journey toward becoming a Robotics Software Engineer specializing in **Perception, Localization, Mapping, and SLAM** using **ROS2** and the **Hiwonder MentorPi A1**.

## Repository Purpose

This repository serves as a self-directed robotics internship. Rather than waiting for an internship opportunity, I am creating my own structured learning experience focused on understanding how robots perceive the world, determine their location, build maps, and navigate autonomously.

The goal is not only to run robotics software, but to understand the engineering principles behind it well enough to explain, debug, document, and improve it.

## Focus Areas

- ROS2 fundamentals
- Robot sensors and sensor data
- Coordinate frames and transforms
- Pose, odometry, and motion estimation
- Mapping
- Localization
- SLAM
- Real robot experiments using the MentorPi A1
- Failure analysis and debugging
- Technical communication through structured documentation

## Hardware Platform

Primary robot platform:

**Hiwonder MentorPi A1 Raspberry Pi Robot Car**  
Ackerman Chassis, ROS2 AI Coding Robot, SLAM and Autonomous Driving Advanced Kit with Raspberry Pi 5 4GB.

## Phase Roadmap

| Phase | Topic | Goal |
|---|---|---|
| Phase 01 | Documentation & Theory | Understand the robot, ROS2, and SLAM at a foundational level. |
| Phase 02 | Pose & Coordinate Frames | Understand robot pose, coordinate systems, and transforms. |
| Phase 03 | Odometry | Understand how wheel motion and sensors estimate robot movement. |
| Phase 04 | Mapping | Understand how sensor data becomes a map. |
| Phase 05 | Localization | Understand how a robot estimates where it is inside a known map. |
| Phase 06 | SLAM | Understand how localization and mapping are solved together. |
| Phase 07 | ROS2 Deep Dive | Study nodes, topics, messages, launch files, TF, and navigation stack concepts. |
| Phase 08 | Real MentorPi Experiments | Run experiments on the physical robot and document results. |
| Phase 09 | Failure Analysis | Document errors, failures, debugging steps, and lessons learned. |

## Engineering Notebook Standards

Each phase should document:

- What I studied
- What I built or tested
- What worked
- What failed
- What I learned
- What I still need to understand
- Screenshots, diagrams, maps, logs, or code when useful

The long-term goal of this notebook is to build a strong public portfolio demonstrating growth toward robotics software engineering, especially in perception, localization, mapping, and SLAM.

This repository is intended to be readable by robotics employers, professors, mentors, and engineering recruiters.
